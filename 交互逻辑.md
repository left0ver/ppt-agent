# PPT Agent 交互逻辑

本文基于当前仓库中的 [agent/app.py](/data3/zwc/ppt-agent/agent/app.py) 和 [agent/modify_ppt.py](/data3/zwc/ppt-agent/agent/modify_ppt.py) 整理，目标是把现有 LangGraph agent 的真实行为翻译成一套适合 `FastAPI + React` 的前后端交互方案。

## 1. 当前 agent 的真实能力

当前项目里有两条链路：

1. 生成链路
   入口在 [agent/app.py](/data3/zwc/ppt-agent/agent/app.py)。
   输入是 `InputSchema.ppt_requirement`，不是单独的 `theme`。
2. 修改链路
   入口在 [agent/modify_ppt.py](/data3/zwc/ppt-agent/agent/modify_ppt.py)。
   输入是 `thread_id + ppt_type + user_instruction`，用于修改已生成的某一页 SVG。

生成链路的顺序如下：

1. 用户输入一句自然语言需求，例如“我是学生，想做一个 DeepSeek 论文汇报，排版使用上下结构”。
2. agent 先抽取 `ppt_info`，然后第一次 `interrupt`，要求用户确认或编辑：
   - `target_audience`
   - `user_role`
   - `num_pages`
   - `theme`
   - `layout_style`
3. 第二次 `interrupt`，询问是否上传 PPT 内容资料，以及可选的 URL 来源。
4. 第三次 `interrupt`，询问是否上传 PPT 模板。
5. 如果用户有内容资料：
   - 解析本地文件
   - 抓取用户提供的 URL
   - 用这些内容生成 `ppt_outline`
6. 如果用户没有内容资料：
   - 调用搜索模型搜索资料
   - 抓取网页内容
   - 生成 `ppt_outline`
7. 按页生成 `ppt_page_contents`
8. 批量生成 `first_draft` SVG
9. 第四次 `interrupt`，询问最终风格，例如“绿色简约风”
10. 基于初稿生成 `final_ppt` SVG
11. 结束

修改链路的顺序如下：

1. 前端选择修改对象：`初稿` 或 `终稿`
2. 用户输入自然语言修改指令
3. agent 判断是否属于“修改 PPT”意图
4. 调用 `modify_ppt` 工具修改指定页 SVG
5. 返回被修改页的新 SVG

## 2. LangGraph 状态与文件产物

生成链路中的核心状态保存在 `State` 中，前后端真正需要关心的是这些字段：

- `ppt_info`
- `have_ppt_content_files`
- `ppt_content_source_urls`
- `have_ppt_template`
- `ppt_content_files_markdown_contents`
- `web_fetch_results`
- `ppt_outline`
- `ppt_page_contents`
- `first_draft_results`
- `user_ppt_style`
- `final_ppt_results`

生成过程中会写入磁盘的关键文件：

```text
user_data/{thread_id}/
  context_files/                       # 用户上传的资料文件
  context_parse/                       # MinerU 解析结果
  template/                            # 用户上传的模板
  first_draft/page_1.svg
  first_draft/page_2.svg
  ...
  final_ppt/page_1.svg
  final_ppt/page_2.svg
  ...
  ppt_content_files_markdown_contents.json
  ppt_outline.json
  ppt_page_contents.json
  first_draft_results.json
  final_ppt_results.json
```

这意味着前端无需等待导出成 `.pptx` 才能预览，后端可以直接把 SVG 列表返回给前端做即时展示。

## 3. 当前实现里必须注意的事实

这些细节会直接影响 API 设计，不能按想象写：

1. 启动参数是 `ppt_requirement`
   [agent/app.py](/data3/zwc/ppt-agent/agent/app.py) 的 `InputSchema` 只定义了 `ppt_requirement`。
2. 第一次 `resume` 需要传完整的 `ppt_info` 对象
   不是只传某个字段。
3. 第二次 `resume` 需要传对象
   结构应为：
   ```json
   {
     "have_ppt_content_files": true,
     "ppt_content_source_urls": ["https://..."]
   }
   ```
4. 第三次 `resume` 只需要传布尔值
   也就是 `have_ppt_template`。
5. 第四次 `resume` 只需要传字符串
   也就是 `user_ppt_style`。
6. 当前模板解析节点实际只查找 `template.ppt` / `template.pptx`
   但中断里给用户展示的是 `pptx` 和 `pdf`。如果前端允许上传 `pdf`，后端需要先修正这段逻辑，或者在文档中明确“暂时仅支持 ppt/pptx 模板参与生成”。
7. `current_timeline` 目前只在 `ask_for_ppt_info` 后被设置为 `INFO_GATHERED`
   后续节点没有持续更新，所以前后端不要直接把它当完整业务状态机。
8. 当前主链路没有流式事件输出
   `agent.invoke(...)` 是阻塞到下一个 interrupt 或结束才返回，因此后端若要展示进度，需要自己封装任务状态。

## 4. 建议的后端职责拆分

FastAPI 不要只是简单透传 LangGraph，而是承担一层“协议适配器”的角色：

1. 会话管理
   生成 `thread_id`，管理会话目录。
2. 文件管理
   接收内容文件和模板文件，保存到 `user_data/{thread_id}/...`。
3. Agent 编排
   负责 `start_workflow`、`resume_workflow`、`modify_ppt_agent.invoke`。
4. 状态投影
   把 LangGraph 的 `__interrupt__`、最终结果、异常，转换成前端稳定可消费的响应结构。
5. 结果聚合
   返回 `outline`、`page_contents`、`first_draft_results`、`final_ppt_results`，以及对应 SVG 预览地址。

## 5. 建议的统一响应格式

前端不要直接消费 LangGraph 原始返回，而是统一由后端转成下面这种结构：

```json
{
  "thread_id": "uuid",
  "status": "interrupted",
  "stage": "awaiting_ppt_info",
  "interrupt": {
    "type": "edit",
    "title": "ppt的相关信息",
    "payload": {
      "target_audience": "导师和同学",
      "user_role": "学生",
      "num_pages": 10,
      "theme": "DeepSeek 论文汇报",
      "layout_style": "top_bottom"
    }
  },
  "data": null,
  "error": null
}
```

生成完成时：

```json
{
  "thread_id": "uuid",
  "status": "completed",
  "stage": "completed",
  "interrupt": null,
  "data": {
    "ppt_outline": {},
    "ppt_page_contents": [],
    "first_draft_results": [],
    "final_ppt_results": []
  },
  "error": null
}
```

失败时：

```json
{
  "thread_id": "uuid",
  "status": "failed",
  "stage": "failed",
  "interrupt": null,
  "data": null,
  "error": {
    "message": "..."
  }
}
```

建议 `stage` 使用后端自定义值，而不是直接暴露 LangGraph 内部字段：

- `awaiting_ppt_info`
- `awaiting_content_sources`
- `awaiting_template`
- `generating_outline`
- `generating_page_contents`
- `generating_first_draft`
- `awaiting_final_style`
- `generating_final_ppt`
- `completed`
- `failed`

## 6. React 前端页面流

建议 React 前端按“单会话工作台”来设计，而不是普通聊天窗口。

### 6.1 页面结构

建议基于 `Ant Design` 的 `Layout` 做一个固定工作台，而不是普通聊天页。

推荐结构如下：

1. 最外层使用 `Layout`
   顶部用 `Header` 放产品标题、当前会话信息、全局操作按钮。
2. 左侧使用 `Sider`
   放步骤导航、会话摘要、提示词快捷入口。
   组件建议用 `Steps + Card + Divider + Prompts`。
3. 中间使用主内容区 `Content`
   根据当前阶段切换不同工作面板，例如需求输入、信息表单、文件上传、生成进度、风格输入。
4. 右侧使用辅助预览区
   如果页面宽度足够，建议单独留一个预览栏；如果宽度不足，则改为主内容区内的 `Tabs` 切换。

更具体的区块建议如下：

1. 顶部栏
   用 `Flex` 或 `Space` 排布标题、`Tag` 状态标记、刷新按钮、重新开始按钮。
2. 左侧流程区
   用 `Steps` 展示：
   - 需求输入
   - 信息确认
   - 资料来源
   - 模板上传
   - 初稿生成
   - 风格确认
   - 终稿完成
3. 左侧辅助区
   用 `Card` 放“推荐提示词”，内部使用 `Ant Design X` 的 `Prompts`。
   再用一个 `Card` 放“当前会话信息”，内部使用 `Descriptions`。
4. 中间主操作区
   用一个主 `Card` 承载当前步骤内容，避免页面频繁跳转。
   聊天式输入和系统反馈混合区域，建议使用 `Bubble.List + Sender`。
   表单类区域，建议使用 `Form`。
5. 右侧预览区
   顶部使用 `Tabs` 切换 `初稿` / `终稿`。
   左边缩略图列表用 `List` 或自定义 `Card` 网格。
   右边当前页大图用 `Card` 包裹 SVG。
   页级操作区用 `Button + Dropdown + Drawer`。

预览区建议支持：

- 使用 `Tabs` 切换初稿 / 终稿
- 使用 `List` 展示按页缩略图
- 使用 `Card` 展示当前页 SVG 大图预览
- 使用 `Drawer` 或 `Modal` 打开单页修改面板
- 使用 `Segmented` 或 `Radio.Group` 切换“缩略图视图 / 单页视图”

### 6.2 前端状态

如果前端采用 `Ant Design X + Ant Design` 的工作台模式，建议把状态分成“会话状态”“表单状态”“预览状态”“交互状态”四类，而不是全部堆在一个组件里。

### 6.2.1 会话状态

这部分驱动 `Layout/Header/Steps/Tag`：

- `threadId`
- `sessionStatus`
  建议取值：`idle | running | interrupted | completed | failed`
- `stage`
  对应当前工作阶段，例如 `awaiting_ppt_info`
- `interrupt`
  后端返回的中断信息
- `error`
- `sessionMeta`
  例如创建时间、最近更新时间、已生成页数

### 6.2.2 表单状态

这部分驱动 `Form`、`Upload`、`Sender`：

- `pptRequirement`
  对应第一次输入框，建议绑定 `Ant Design X` 的 `Sender`
- `pptInfoDraft`
  对应 `PptInfoForm` 的表单草稿
- `contentFiles`
  已上传的资料文件列表，驱动 `Upload` 的 `fileList`
- `contentUrls`
  URL 列表，驱动 `Form.List`
- `templateFile`
  已上传模板文件
- `finalStyle`
  最终风格输入值
- `modifyInstruction`
  当前单页修改输入值

### 6.2.3 预览状态

这部分驱动 `Tabs`、`List`、SVG 预览卡片：

- `firstDraftResults`
- `finalDraftResults`
- `previewPptType`
  建议取值：`first_draft | final_ppt`
- `selectedPage`
- `selectedSvgContent`
- `outlineData`
- `pageContents`

### 6.2.4 交互状态

这部分驱动 `Spin`、`Drawer`、`Modal`、`Message`、`Notification`：

- `isStarting`
- `isSubmittingPptInfo`
- `isUploadingContentFiles`
- `isUploadingTemplate`
- `isGenerating`
- `isSubmittingFinalStyle`
- `isModifyingPage`
- `isModifyDrawerOpen`
- `messages`
  用于 `Ant Design X` 的 `Bubble.List`，保存系统提示、用户输入、关键结果消息

### 6.2.5 推荐状态组织方式

推荐把状态拆成几个 hook 或 store：

1. `useSessionStore`
   管理 `threadId/sessionStatus/stage/interrupt/error`
2. `usePptFormStore`
   管理 `pptRequirement/pptInfoDraft/contentFiles/contentUrls/templateFile/finalStyle`
3. `usePptPreviewStore`
   管理 `firstDraftResults/finalDraftResults/selectedPage/previewPptType`
4. `useUiStateStore`
   管理各种 loading、drawer、modal、notification 状态

如果项目规模不大，可以先用 `React useState + useReducer`；
如果后续要接轮询、WebSocket、缓存恢复，建议直接使用 Zustand。

### 6.3 用户主流程

下面按用户实际操作顺序说明，并明确每一步建议使用的 `Ant Design X + Ant Design` 组件。

1. 用户进入页面，前端先创建会话。
   建议使用 `Layout` 搭建整体工作台，顶部用 `PageHeader` 或页面标题区展示当前会话，左侧流程区使用 `Steps` 展示当前阶段。
2. 用户输入 `ppt_requirement`。
   输入区建议使用 `Ant Design X` 的 `Sender` 作为主输入组件，底部配合 `Prompts` 展示若干示例需求，例如“论文汇报”“产品介绍”“培训课件”。
3. 后端启动 agent，返回 `awaiting_ppt_info`。
   会话消息流建议使用 `Ant Design X` 的 `Bubble` 或 `Bubble.List` 展示“已收到需求，正在分析”这类系统反馈；页面整体加载态可以用 `Spin` 或 `Skeleton`。
4. 前端渲染 `ppt_info` 表单，允许用户修改后提交。
   这一段建议使用 `Card + Form + Input + InputNumber + Select + Button`。
   `target_audience`、`user_role`、`theme` 用 `Input`，`num_pages` 用 `InputNumber`，`layout_style` 用 `Select`。
5. 后端恢复流程，返回 `awaiting_content_sources`。
   左侧 `Steps` 更新到“资料来源”，主区域可继续用 `Bubble` 展示“请上传资料或补充 URL”。
6. 前端展示资料上传与来源输入。
   文件上传建议使用 `Upload.Dragger`，URL 输入建议使用 `Form.List + Input` 支持多条 URL。
   “跳过资料”使用 `Button`，已上传文件列表使用 `List` 或 `Table`。
7. 后端恢复流程，返回 `awaiting_template`。
   这一步建议仍放在一个 `Card` 中，并在顶部用 `Alert` 说明模板是可选项。
8. 前端展示模板上传或跳过。
   模板上传使用 `Upload.Dragger`，跳过使用 `Button`，已上传模板信息可用 `Descriptions` 展示文件名、大小、类型。
9. 提交后进入长任务阶段，前端切换到“生成中”界面。
   这一阶段建议使用 `Steps + Progress + Result` 组合。
   `Steps` 展示当前进行到“大纲生成 / 内容生成 / 初稿生成”，中间区域可用 `Spin`，如果后端提供细粒度状态，可在右侧使用 `Timeline` 展示日志。
10. 后端完成初稿生成后，agent 再次中断，返回 `awaiting_final_style`，同时附带 `first_draft_results`。
    页面主区域建议切换到“初稿预览工作台”。
    左侧缩略图使用 `List` 或自定义卡片列表，右侧大图区域使用 `Card` 包裹 SVG 预览。
11. 前端展示初稿预览，并让用户输入最终风格。
    风格输入建议使用 `Card + Form + AutoComplete + Input`，预设风格可以通过 `AutoComplete` 提供，例如“绿色简约风”“黑色科技风”“蓝白商务风”。
    若想更接近对话式体验，可以在预览区下方增加 `Ant Design X` 的 `Sender` 作为风格输入框。
12. 提交风格后，后端恢复流程，生成终稿。
    终稿生成阶段继续沿用 `Steps + Progress + Spin`。
    如果希望用户不离开预览页，可以在右上角用 `Badge` 标记“生成中”状态。
13. 前端展示终稿预览，并开放单页修改功能。
    整体建议使用 `Tabs` 在“初稿 / 终稿”之间切换。
    缩略图区用 `List`，大图区用 `Card`，单页修改面板用 `Drawer` 或 `Modal`。
    修改输入建议使用 `Ant Design X` 的 `Sender`，配合 `Bubble.List` 展示修改历史；提交后局部刷新对应页 SVG。

## 7. FastAPI API 设计

下面的接口设计是围绕当前 agent 的实际输入输出组织的。

### 7.1 创建会话

`POST /api/sessions`

响应：

```json
{
  "thread_id": "uuid"
}
```

### 7.2 启动 PPT 生成

`POST /api/ppt/start`

请求：

```json
{
  "thread_id": "uuid",
  "ppt_requirement": "我是一个学生，我想给导师做一个有关 deepseek 的论文汇报，排版使用上下结构"
}
```

后端处理：

1. 调用 `start_workflow(InputSchema(ppt_requirement=...), thread_id)`
2. 如果收到 `__interrupt__`，转换为统一响应
3. 如果已完成，返回结果

### 7.3 提交 PPT 基本信息

`POST /api/ppt/resume/ppt-info`

请求：

```json
{
  "thread_id": "uuid",
  "ppt_info": {
    "target_audience": "导师和同学",
    "user_role": "学生",
    "num_pages": 10,
    "theme": "DeepSeek 论文汇报",
    "layout_style": "top_bottom"
  }
}
```

后端处理：

1. `resume_workflow(thread_id, ppt_info)`
2. 返回下一个 interrupt 或进入生成流程

### 7.4 上传内容资料文件

`POST /api/ppt/content-files`

`multipart/form-data`

字段：

- `thread_id`
- `files[]`

保存路径：

- `user_data/{thread_id}/context_files/`

响应中返回文件列表即可。

### 7.5 提交内容来源确认

`POST /api/ppt/resume/content-sources`

请求：

```json
{
  "thread_id": "uuid",
  "have_ppt_content_files": true,
  "ppt_content_source_urls": [
    "https://example.com/a",
    "https://example.com/b"
  ]
}
```

后端需要把这两个字段组装成一个对象，再调用：

```python
resume_workflow(
    thread_id,
    {
        "have_ppt_content_files": have_ppt_content_files,
        "ppt_content_source_urls": ppt_content_source_urls,
    },
)
```

### 7.6 上传模板文件

`POST /api/ppt/template`

`multipart/form-data`

字段：

- `thread_id`
- `file`

保存路径：

- `user_data/{thread_id}/template/template.pptx`
  或
- `user_data/{thread_id}/template/template.ppt`

如果保留当前 agent 逻辑，建议前端只允许上传 `ppt/pptx`。
如果要支持 `pdf`，需要先改 [agent/app.py](/data3/zwc/ppt-agent/agent/app.py) 里 `parser_ppt_template` 的查找逻辑。

### 7.7 提交模板是否使用

`POST /api/ppt/resume/template`

请求：

```json
{
  "thread_id": "uuid",
  "have_ppt_template": true
}
```

后端调用：

```python
resume_workflow(thread_id, True)
```

### 7.8 提交最终风格

`POST /api/ppt/resume/final-style`

请求：

```json
{
  "thread_id": "uuid",
  "user_ppt_style": "绿色简约风"
}
```

后端调用：

```python
resume_workflow(thread_id, "绿色简约风")
```

### 7.9 查询会话状态

`GET /api/ppt/status?thread_id=uuid`

返回当前业务状态、已有结果摘要、可预览页数等。

这个接口的状态建议由后端自己维护，不建议直接依赖 `get_status()` 的 `current_timeline`。

### 7.10 获取某页 SVG

`GET /api/ppt/svg/{thread_id}/{ppt_type}/{page}`

其中：

- `ppt_type` 为 `first_draft` 或 `final_ppt`
- `page` 为页码

用于前端按页预览。

### 7.11 修改单页 PPT

`POST /api/ppt/modify`

请求：

```json
{
  "thread_id": "uuid",
  "ppt_type": "终稿",
  "user_instruction": "第3页背景改成蓝色，标题改粗一点"
}
```

后端处理：

1. 调用 `modify_ppt_agent.invoke(...)`
2. 返回：
   - `response_content`
   - `new_svg_list`

响应示例：

```json
{
  "thread_id": "uuid",
  "status": "completed",
  "data": {
    "response_content": "第3页的终稿修改成功.",
    "new_svg_list": [
      {
        "page": 3,
        "new_svg_content": "<svg ...>...</svg>"
      }
    ]
  }
}
```

## 8. 生成链路的前后端时序

### 8.1 主生成流程

1. React 调 `POST /api/sessions`
2. React 调 `POST /api/ppt/start`
3. FastAPI 调 `start_workflow(...)`
4. LangGraph 返回第一次 `interrupt`
5. FastAPI 转成 `awaiting_ppt_info`
6. React 渲染 `ppt_info` 表单
7. React 调 `POST /api/ppt/resume/ppt-info`
8. FastAPI 调 `resume_workflow(thread_id, ppt_info)`
9. LangGraph 返回第二次 `interrupt`
10. FastAPI 转成 `awaiting_content_sources`
11. React 上传文件，填写 URL
12. React 调 `POST /api/ppt/resume/content-sources`
13. LangGraph 返回第三次 `interrupt`
14. React 上传模板或跳过
15. React 调 `POST /api/ppt/resume/template`
16. FastAPI 进入实际生成
17. 生成初稿后，LangGraph 返回第四次 `interrupt`
18. FastAPI 转成 `awaiting_final_style`，同时附带初稿结果
19. React 预览初稿并输入最终风格
20. React 调 `POST /api/ppt/resume/final-style`
21. LangGraph 生成终稿并结束
22. FastAPI 返回终稿结果
23. React 进入可修改态

### 8.2 单页修改流程

1. React 选择 `初稿` 或 `终稿`
2. React 选择页码并输入修改指令
3. React 调 `POST /api/ppt/modify`
4. FastAPI 调 `modify_ppt_agent.invoke(...)`
5. 返回新 SVG
6. React 局部更新对应页预览

## 10. 后端实现建议

### 10.1 不要在请求线程里裸跑长任务

当前 `agent.invoke(...)` 会阻塞较久。更稳妥的方式有两种：

1. 简单版本
   先同步调用，前端用 loading 等待。
   适合本地开发。
2. 正式版本
   后端创建后台任务，前端轮询 `/api/ppt/status` 或走 WebSocket/SSE。
   这样更适合多页生成场景。

### 10.2 建议维护自己的 SessionStore

至少记录：

- `thread_id`
- `status`
- `stage`
- `last_interrupt`
- `error`
- `created_at`
- `updated_at`

可以先放内存，后续再切换 Redis/数据库。

### 10.3 对 LangGraph 返回做统一适配

建议封装：

- `adapt_agent_response(response, thread_id) -> ApiResponse`
- `extract_interrupt(response) -> stage`
- `collect_generated_assets(thread_id) -> data`

这样 FastAPI 层会清晰很多。

## 11. 当前代码与前后端实现之间的差距

在正式写 FastAPI 和 React 之前，建议先修正这些问题：

1. 模板格式不一致
   中断提示允许 `pdf`，但模板解析只处理 `ppt/pptx`。
2. 时间线字段不完整
   `current_timeline` 不能覆盖完整业务阶段。
3. 缺少统一响应模型
   当前返回还是 LangGraph 原始结构。
4. 缺少静态文件访问或 SVG 读取接口
   前端目前无法稳定按路径预览所有页面。
5. 缺少任务中状态
   当前只能“等待返回”，无法精细展示“正在生成第几步”。
6. 修改链路和生成链路尚未统一到同一套会话 API 下
   React 侧需要统一会话上下文。

## 12. 推荐落地顺序

建议实现顺序如下：

1. 先封装 `FastAPI` 会话接口和统一响应结构
2. 再实现文件上传和 SVG 预览接口
3. 然后接入主生成链路的 4 次 `interrupt/resume`
4. 接着做终稿预览和单页修改
5. 最后再补轮询、取消生成、WebSocket/SSE 等增强功能

按这个顺序做，可以先把核心闭环跑通，再处理体验层问题。
